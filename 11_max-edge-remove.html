Q11. 删点后图中最大边权（无直接 LC）

来源：SWE II L3 R1 · LC: 无直接 LC ❓ 无对应 · 难度: Med

Graph Preprocessing Heap ❓ 无对应 Med

题目概览

来源：Google SWE II L3 R1 原描述：加权图，给一个节点，删除它和它所有的边，返回剩下图中最大边权。讨论了 priority queue、multiset、preprocessing 三种思路。

题目

EN: Given a weighted graph and a node v, remove v and all its incident edges. Return the maximum edge weight in the remaining graph. Support multiple such queries.

中文：加权图，给定节点 v，删除 v 和它所有相邻边，返回剩下的最大边权。支持多次查询。

澄清问题

  • EN: How many queries? / Are queries on the same graph (each query independent — node not actually removed) or sequential (each query mutates the graph)? / Are weights unique? / What if graph becomes empty?
  • 中：查询次数？每次查询是独立的（图不变，只是"假设删了 v"）还是累积的（真的删了）？边权是否唯一？图被删空怎么办？

🧠 整体思路

面经的强烈推荐路径：brute force → 优化 → 最优。这道题的演进过程比答案本身更重要。

4 种思路（从笨到聪明）： 1. Brute force：每次查询都遍历所有边，跳过连 v 的，求 max。每次 O(E)，多查询 O(Q·E)。 2. Max-heap：边按权重入堆，pop 直到顶不连 v；但 pop 出来的下次查询还要还原，难。 3. 排序边一次：所有边按权重降序排好；每次查询从前往后扫，第一条不连 v 的就是答案。 4. 预处理：对每个节点预计算"删它之后的最大边"，存到字典；查询 O(1)。

关键洞察：边按权重降序后，"删 v 后的最大边" = 第一条不接触 v 的边。

适用场景： - 单次查询 → 思路 3 足够。 - 多次重复查询 → 思路 4 预处理。 - 大量节点查询不同 → 思路 4 的 O(V·E) 预处理 vs O(Q·E) 在线，看哪个划算。

🪜 思路拆解

Step 1 · 数据结构选择 - edges = [(u, v, w), ...]：边列表。 - 一次性 sorted(edges, key=lambda e: -e[2])：按权重降序。

Step 2 · 思路 3 — 排序后扫描 - 排好序后，对每次查询 query(node)： - 从前往后扫边，找第一条 u != node and v != node 的。 - 返回它的权重；扫不到返回 -1。 - 复杂度：单次 O(E)，总 O(Q·E)。

Step 3 · 思路 4 — 预处理到 O(1) query - 遍历所有节点 set，对每个节点跑一次思路 3。 - 把答案缓存到 self.ans[node]。 - 复杂度：build O(V·E)，query O(1)。 - 空间：O(V) 缓存。

Step 4 · 进一步优化（如果面试官追问） - 思路 4 的 build 可以优化：我们只关心前 max_degree + 1 条边即可（因为任何节点最多 "屏蔽" degree 条边，第 degree+1 条边必然能用）。 - 复杂度降到 build O(E + V·min(E, max_degree+1))。

Step 5 · Edge cases - 节点不存在 → 退化到"返回全图最大边"。 - 节点孤立无边 → 不影响其他边的存在，答案就是全图最大边。 - 多重边：可能有，但不影响算法 — 边按独立条目处理。

⌨️ 逐步实现

Step 1-2 · 思路 3：排序 + 扫描

from collections import defaultdict

class GraphQuery:
    def __init__(self, n, edges):
        self.all_edges = sorted(edges, key=lambda e: -e[2])  # 一次降序

    def max_weight_after_remove(self, node):
        for u, v, w in self.all_edges:
            if u != node and v != node:
                return w
        return -1

Step 3 · 思路 4：预处理到 O(1)

class GraphQueryFast:
    def __init__(self, n, edges):
        sorted_edges = sorted(edges, key=lambda e: -e[2])
        nodes = set()
        for u, v, _ in edges:
            nodes.add(u)
            nodes.add(v)

        self.ans = {}
        for node in nodes:
            self.ans[node] = -1
            for u, v, w in sorted_edges:
                if u != node and v != node:
                    self.ans[node] = w
                    break

    def query(self, node):
        return self.ans.get(node, -1)

Python 完整实现（预处理法 — Final Solution from 面经）

from collections import defaultdict


class GraphQuery:
    """假设：每次查询独立，预处理后 O(1) 回答。"""

    def __init__(self, n, edges):
        self.adj = defaultdict(list)  # node -> sorted list of (weight, other_end)
        for u, v, w in edges:
            self.adj[u].append((w, v))
            self.adj[v].append((w, u))
        # 全图所有边按权重降序排序
        self.all_edges = sorted(edges, key=lambda e: -e[2])

    def max_weight_after_remove(self, node):
        for u, v, w in self.all_edges:
            if u != node and v != node:
                return w
        return -1  # 全部被删光

上面是 O(query × E) — 没真正"预处理到 O(1)"。下面是 O(1) per query。

O(1) per query 版本

class GraphQuery:
    def __init__(self, n, edges):
        edges_sorted = sorted(edges, key=lambda e: -e[2])
        # 对每个 node v，缓存"不含 v 的最大边"
        self.ans = {}
        # 朴素 O(N·E)
        nodes = set()
        for u, v, _ in edges:
            nodes.add(u); nodes.add(v)
        for node in nodes:
            self.ans[node] = -1
            for u, v, w in edges_sorted:
                if u != node and v != node:
                    self.ans[node] = w
                    break

    def query(self, node):
        return self.ans.get(node, edges_sorted[0][2] if edges_sorted else -1)

讲解策略

  1. 递进 3 个方案，正是面经的强烈推荐路径： - Brute: 每次删 + 扫所有边求 max，O(E) per query。 - Max-heap: 扔进堆，pop 直到顶不连 v；问题是 pop 出来的还要还原。 - Multiset / sorted edges: 一次排序，每次查询扫排序后的边，O(E) 但常数小。 - Preprocessing: 对每个 node 预计算答案，每次查询 O(1)。
  2. 关键洞察：所有边按权重降序排序，第一条不连 v 的就是答案。
  3. 如果允许超大 N：可以只对前 K 大的边做查询，K = min(E, max_degree + 1) 就够。
  4. 复杂度对比：Brute O(Q·E)，预处理 O(V·E) build + O(1) query。
  5. 如果是 streaming（边一条条来）：维护一个 max-heap + lazy deletion。

常见坑

  • "删除"的含义不问清楚：每次独立 vs 累积，决定算法完全不同。
  • 多重边：节点间多条边时，"删 v 的所有边"会一次删多条。
  • 节点不存在：早些问，避免边界 crash。
  • 直接说 max-heap：能跑通但 follow-up 问"优化"时容易卡，预处理才是 strong-hire 答案。
