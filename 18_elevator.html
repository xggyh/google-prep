18. 电梯调控器（OOP 设计）

来源：L6 R1 · LC: OOP 设计 🟡 变体 · 难度: Hard

OOP Design State Machine SCAN 🟡 变体 Hard

题目概览

来源：Google L6 Onsite Coding 1 原描述：输入是用户按电梯的时间、所在楼层、要去的楼层。需要自己定义输出、类、接口。讨论公平优先、路径优先、控制多台电梯。

题目

EN: Design an elevator controller. Input: a stream of requests (time, source_floor, target_floor). Output: design the schedule. Discuss fairness vs efficiency (SCAN/LOOK), multi-car coordination.

中文：设计电梯调度系统。输入：(时间, 起始楼层, 目的楼层) 的请求流。自己定义输出、类、API。讨论公平 vs 路径优先、多台电梯协同。

澄清问题（这题超 open，澄清是核心！）

  • EN:
  • Single elevator or multiple? How many cars?
  • Capacity per car?
  • Floors range?
  • Is the request "I'm at floor 3, going to floor 7" all at once, or two-step (call button + destination)?
  • Optimize for: average wait time? Max wait? Throughput?
  • Online (real-time) or offline (all requests known)?
  • Are requests preemptible (can re-route mid-trip)?
  • What's the cost model (time per floor, door open time)?
  • 中：
  • 几台电梯？每台容量？
  • 楼层数？
  • 请求格式（一次性给出源/目还是两步：呼叫 + 进门后按目的层）？
  • 优化目标（平均等待 / 最大等待 / 吞吐量）？
  • 在线还是离线？
  • 中途能改变路线吗？
  • 时间成本模型（每层多少秒、开关门多少秒）？

🧠 整体思路

L6 这题原帖说"题目过于 open，clarify 时间长"。正确做法是把 clarify 当成主菜，不是开胃菜。

经典电梯调度算法叫 SCAN / LOOK： - 电梯保持一个方向（UP / DOWN / IDLE）。 - UP 时一直向上停所有需要上行的楼层，到顶（或没人）后掉头 DOWN。 - 类似硬盘磁头扫描算法。

多梯调度的核心：dispatch 时算每辆梯的"成本"，选最小的。成本包括距离 + 方向匹配 + 当前负载。

两个评估指标（先和面试官敲定）： - 公平 (fairness)：FCFS 严格按时间顺序服务。等待方差小，但平均等待时间长。 - 效率 (efficiency)：SCAN / LOOK 顺路服务。平均等待短，但可能有人长时间等待（饿死）。

OOP 拆 3 层： - Request：纯数据。 - Elevator：单梯状态机。 - ElevatorBank：多梯调度器。

🪜 思路拆解（必须先 clarify！）

Step 0 · 疯狂 clarify（至少 4-5 个问题） - 几台电梯？容量？ - 楼层数？ - 请求格式：一次性 (源, 目)，还是两步（呼叫 + 进梯后按目的层）？ - 优化目标：平均等待 / 最大等待 / 吞吐？ - 在线还是离线？预先知道所有请求 vs 流式来。 - 时间模型：每层多少秒？开门多少秒？

Step 1 · 定义 Direction Enum - UP=1, DOWN=-1, IDLE=0。 - 用数字便于楼层加减。

Step 2 · 定义 Request - 三个字段：time, source, target。

Step 3 · 单梯 Elevator 类的状态 - id, cur_floor, direction, up_stops (set), down_stops (set), passengers (list)。 - 把待停楼层分上行 / 下行两个 set — 让 SCAN 算法逻辑清楚。

Step 4 · Elevator 的 API - add_request(req)：根据 source 相对 cur_floor 的位置分到 up_stops 或 down_stops。 - step()：走一格 — 处理当前层 + 决定下一步方向。 - _handle_floor()：到达某层时开门 / 接客 / 出客。

Step 5 · SCAN 方向决策 - 当前 UP： - up_stops 还有更高的 → 继续上。 - 否则 down_stops 有 → 转 DOWN。 - 都没 → IDLE。 - 当前 DOWN：对称。 - 当前 IDLE：看哪边有 stops 就走那边。

Step 6 · 多梯 ElevatorBank - dispatch(req)：算每辆梯的成本，选最低。 - 成本函数（面试时主动讲清楚为什么这么算）： - 距离 |car.floor - req.source| 是基础。 - 方向相符加分（如电梯正在 UP 且 req 也是 UP，直接顺路）。 - 反向大幅惩罚。 - 高 load 时减分。 - step_all()：每个 tick 让所有电梯走一格。

Step 7 · 持续 dry run - 例子：3 个请求 + 2 辆梯，走 5 个 tick，把每辆梯的 (floor, direction, up_stops, down_stops) 表格画出来。

Step 6 · ElevatorBank 调度

class ElevatorBank:
    def __init__(self, num_cars, num_floors):
        self.cars = [Elevator(i, num_floors) for i in range(num_cars)]

    def dispatch(self, req):
        best = min(self.cars, key=lambda c: self._cost(c, req))
        best.add_request(req)
        return best.id

    def _cost(self, car, req):
        d = abs(car.cur_floor - req.source)
        # 顺路 = 同向 + req 在前方
        if car.direction == Direction.UP   and req.source >= car.cur_floor and req.target > req.source:
            return d
        if car.direction == Direction.DOWN and req.source <= car.cur_floor and req.target < req.source:
            return d
        if car.direction == Direction.IDLE:
            return d
        return d + 100                        # 反向重罚

    def step_all(self):
        for car in self.cars:
            car.step()

Python 完整实现（单梯 SCAN 算法）

from enum import Enum
from collections import deque


class Direction(Enum):
    UP = 1
    DOWN = -1
    IDLE = 0


class Request:
    def __init__(self, time, source, target):
        self.time = time
        self.source = source
        self.target = target


class Elevator:
    def __init__(self, eid, num_floors):
        self.id = eid
        self.cur_floor = 0
        self.direction = Direction.IDLE
        self.up_stops = set()    # 待停的上行楼层
        self.down_stops = set()  # 待停的下行楼层
        self.passengers = []     # 已上车，未下车
        self.num_floors = num_floors

    def add_request(self, req):
        # 把 source 加入对应方向的 stop set
        if req.source > self.cur_floor:
            self.up_stops.add(req.source)
        elif req.source < self.cur_floor:
            self.down_stops.add(req.source)
        else:
            # 直接上车
            self._board(req)

    def _board(self, req):
        self.passengers.append(req)
        if req.target > self.cur_floor:
            self.up_stops.add(req.target)
        else:
            self.down_stops.add(req.target)

    def step(self):
        """走一步：一层或处理当前层。"""
        # 处理当前层的下车 / 上车
        self._handle_floor()

        # 决定方向
        if self.direction == Direction.UP:
            if self.up_stops and max(self.up_stops) >= self.cur_floor:
                self.cur_floor += 1
            elif self.down_stops:
                self.direction = Direction.DOWN
                self.cur_floor -= 1
            else:
                self.direction = Direction.IDLE
        elif self.direction == Direction.DOWN:
            if self.down_stops and min(self.down_stops) <= self.cur_floor:
                self.cur_floor -= 1
            elif self.up_stops:
                self.direction = Direction.UP
                self.cur_floor += 1
            else:
                self.direction = Direction.IDLE
        else:  # IDLE
            if self.up_stops:
                self.direction = Direction.UP
            elif self.down_stops:
                self.direction = Direction.DOWN

    def _handle_floor(self):
        if self.direction == Direction.UP and self.cur_floor in self.up_stops:
            self.up_stops.discard(self.cur_floor)
            # 让到这一层的下车 / 上车
        elif self.direction == Direction.DOWN and self.cur_floor in self.down_stops:
            self.down_stops.discard(self.cur_floor)


class ElevatorBank:
    def __init__(self, num_cars, num_floors):
        self.cars = [Elevator(i, num_floors) for i in range(num_cars)]

    def dispatch(self, req):
        """选择最合适的电梯。"""
        best = min(self.cars, key=lambda c: self._cost(c, req))
        best.add_request(req)
        return best.id

    def _cost(self, car, req):
        # 简单成本：距离 + 方向相符加分
        d = abs(car.cur_floor - req.source)
        if car.direction == Direction.UP and req.source >= car.cur_floor and req.target > req.source:
            return d
        if car.direction == Direction.DOWN and req.source <= car.cur_floor and req.target < req.source:
            return d
        if car.direction == Direction.IDLE:
            return d
        return d + 100  # 反向惩罚

    def step_all(self):
        for car in self.cars:
            car.step()

讲解策略（这题 L6 评论说"过于 open"）

  1. 花前 5 分钟疯狂 clarify：上面那 8 个问题至少问 4-5 个。如果不澄清直接写代码 = 必挂。
  2. 明确目标函数：跟面试官敲定优化哪个指标，再选算法。
  3. 从简单开始：先单梯 SCAN（楼宇电梯通用），再扩到多梯。
  4. OOP 清晰： - Request：数据 - Elevator：单梯状态机 - ElevatorBank：调度器 - Direction：枚举
  5. 多梯协同：哪个电梯接哪个请求？给一个 cost function — 距离 + 方向匹配 + load。
  6. 持续 dry run：3 个请求，2 台梯，走一遍每个 step。
  7. L6 的核心是 trade-off discussion：fairness（先来先服务，FCFS）vs efficiency（路径优先，SCAN/LOOK） — 主动讨论两个目标的张力。

常见坑

  • 不澄清直接写代码：这是 L6 这题的最大坑，面经里面试官自己都说"题目过于 open"。
  • 状态机搞复杂：电梯只有 3 个 direction，状态转移要画图。
  • 多梯只是复制单梯：要讨论协同 — 谁接谁的请求？
  • 不考虑容量限制：现实电梯有 max 人数，最好问清楚。
  • 没讲 fairness vs efficiency：L6 必考点。
