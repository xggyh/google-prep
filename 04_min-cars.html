Q04. 租车最少辆数（LC 253 Meeting Rooms II 变体）

来源：L4 SWE AI/ML R2 · LC: LC 253 ✅ 高置信 · 难度: Med

Sweep Line Heap OOP ✅ 高置信 Med

题目概览

来源：Google L4 SWE AI/ML 全套过经 R2 原描述：一个 list of car rental requests，每个有 pickup 和 return time，给 Car class 有 id 和 rental_record=[]。求最少需要多少辆车，并记录每辆车的 assignment。Follow-up：sharded logs across servers/regions 怎么处理。

题目

EN: Given car rental requests [(pickup_time, return_time), ...]. Compute minimum number of cars needed. Then assign each request to a specific Car instance and append to car.rental_record. Follow-up: requests come from logs sharded across many servers/regions — how to handle?

中文：一组租车请求 (取车时间, 还车时间)。求最少需要多少辆车，并把每个请求分配到具体的 Car 对象，更新其 rental_record。Follow-up：日志分片到多服务器/多地区，如何处理？

澄清问题

  • EN: Is return_time exclusive (car can be re-rented at the same instant)? / Are requests sorted? / Can a car be used by multiple regions? / For shard follow-up: do we need exact answer or approximation? Real-time or batch?
  • 中：还车时间是开区间吗（同一时刻能立即出租）？输入是否已排序？是否要求 stable assignment（同样输入给同样答案）？Follow-up：分片是要精确解还是近似？实时还是批处理？

🧠 整体思路

这道题 = LC 253 Meeting Rooms II + 多记一份 assignment。

两种典型解法： - 扫描线：把 (pickup, +1) 和 (return, -1) 排序遍历，累加得"任意时刻同时占用车数"的最大值。简单但没法分配到具体车。 - min-heap：按 pickup 排序请求，堆里维护"已有车的下次空闲时间"。新请求来时看堆顶最早的车是否空了，空了就复用，不空就新开。能记录 assignment。

这题考点是后者 — 因为题目明确要把请求分配到 Car.rental_record。

贪心正确性：每来新请求，复用"最早空闲且 ≤ 当前 pickup"的车一定不亏 — 因为新车也只能服务这个请求，复用相当于省了一辆。

复杂度：O(n log n)，排序 + 堆操作。

🪜 思路拆解

Step 1 · 设计 Car 数据结构 - 字段：id（唯一标识）、rental_record（list of (pickup, return)）。 - 这就是题目原话给的接口，照抄。

Step 2 · 排序所有请求 - 按 pickup_time 升序排。 - 为什么：保证我们按时间推进，堆里"已有车"的状态总是基于"过去"。

Step 3 · 用 min-heap 维护可复用的车 - 堆元素：(next_available_time, car_id, car_ref)。 - next_available_time 是这辆车上次的 return 时间。 - 为什么加 car_id 作 tiebreaker：Python 的 heapq 比较元组时，相等就比下一项，Car 对象不支持比较会抛 TypeError。

Step 4 · 主循环：复用 or 新开 - 取下一个请求 (pickup, ret)。 - 检查堆顶 pool[0][0] <= pickup： - 是 → pop 出来，复用这辆车。 - 否 → 新建 Car，加入车队。 - 把这次租用记录到 car.rental_record。 - 把车以新的 ret 时间 push 回堆。

Step 5 · 返回结果 - 返回 (总车数, cars 列表)，其中每辆 car.rental_record 里都有完整的分配记录。

Step 6 · Follow-up 提示（口头讨论即可） - 还车时间 vs 取车时间是否能相等？必须先问（决定 <= 还是 <）。 - 分片日志（shard across servers）：可以提中央汇聚 + 时间排序，或时间窗分桶 + map-reduce。

⌨️ 逐步实现

Step 1 · Car 类

import heapq

class Car:
    def __init__(self, car_id):
        self.id = car_id
        self.rental_record = []

Step 2-5 · 主算法

def min_cars(requests):
    requests.sort(key=lambda r: r[0])           # Step 2: 按 pickup 排序
    pool = []                                    # min-heap: (return_time, car_id, Car)
    cars = []
    next_id = 0

    for pickup, ret in requests:
        # Step 4: 复用 or 新开
        if pool and pool[0][0] <= pickup:
            _, _, car = heapq.heappop(pool)      # 复用最早空闲的车
        else:
            car = Car(next_id)
            cars.append(car)
            next_id += 1

        car.rental_record.append((pickup, ret)) # 记录分配
        heapq.heappush(pool, (ret, car.id, car)) # 重新入堆

    return len(cars), cars                       # Step 5

Python 完整实现

import heapq


class Car:
    def __init__(self, car_id):
        self.id = car_id
        self.rental_record = []


def min_cars(requests):
    requests.sort(key=lambda r: r[0])
    # 堆: (车的下次可用时间, car_id, Car 对象)
    pool = []
    cars = []
    next_id = 0
    for pickup, ret in requests:
        if pool and pool[0][0] <= pickup:
            _, _, car = heapq.heappop(pool)
        else:
            car = Car(next_id)
            cars.append(car)
            next_id += 1
        car.rental_record.append((pickup, ret))
        heapq.heappush(pool, (ret, car.id, car))
    return len(cars), cars

讲解策略

  1. 先想扫描线：把所有 (pickup, +1) 和 (return, -1) 排序，遍历过程中维护"当前在用的车数"的最大值 — 这只能算车数，无法分配。
  2. 改用 min-heap：堆里存"每辆已存在车的还车时间"，新请求来时看堆顶最早的车是否已还（pickup ≥ heap_top.return），是就复用，否就新开。
  3. 关键决策：复用 vs 新开车的判定 — 必须明确说为什么贪心是对的（交换论证）。
  4. 复杂度：O(n log n)。
  5. Follow-up（sharding）：可以提到这几条思路 — (a) 中央协调器收集所有 region 的 (pickup, return) 事件，仍按时间排序统一处理；(b) 如果只要 count，可以用 sketch / Bloom 但精度不够，租车场景需要精确；(c) 时间窗分桶 + map-reduce。

常见坑

  • 用 sorted set 但 Python 没原生：必须用 heapq，注意 tuple 比较时 Car 对象不可比，所以加 car.id 作 tiebreaker。
  • <= vs <：还车时间到了能不能立刻再租？务必先问。
  • 以为只要 count：题目明确说"要 record 到每辆车"，扫描线不够。
  • Follow-up 答得太工程：Google 想听算法层面的权衡（精确 vs 近似，consistency），别只讲 Kafka/Redis 这类工具。
