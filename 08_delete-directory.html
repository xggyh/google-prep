08. 删除目录递归实现（DFS 文件系统）

来源：L4 两轮 Onsite R2 · LC: LC 1233 思想 🟡 变体 · 难度: Med

DFS Recursion File System 🟡 变体 Med

题目概览

来源：Google L4 两轮 Onsite R2 原描述：给定 getChildrenPaths(path) 和 deleteFile(path) 两个 API，实现 deleteDirectory(path)。问了很多不同实现的 tradeoff。

题目

EN: Given APIs getChildrenPaths(path: str) -> List[str] and deleteFile(path: str) -> None, implement deleteDirectory(path: str) -> None that recursively removes everything under path.

中文：给定两个 API getChildrenPaths(path)（返回子节点路径）和 deleteFile(path)（删除单个文件），实现 deleteDirectory(path) — 递归删除整个目录。

澄清问题

  • EN: How to distinguish a file from a directory? (getChildrenPaths returns empty for files?) / Is deleteFile synchronous? / Can it fail? / Are concurrent modifications happening? / Is the tree depth bounded? / Should we delete the root path itself or just its contents?
  • 中：怎么区分文件和目录（API 怎么告诉我们）？deleteFile 是同步的吗？会失败吗？删除过程中树会被并发修改吗？树深度有限制吗？是删 path 自身还是只删 path 下的内容？

Python 实现

def deleteDirectory(path):
    """假设：getChildrenPaths 返回空表示是文件。"""
    children = getChildrenPaths(path)
    if not children:
        deleteFile(path)
        return
    for child in children:
        deleteDirectory(child)
    deleteFile(path)  # 删自己（如果 API 要求）


def deleteDirectory_iter(path):
    """迭代版，避免深递归爆栈。后序遍历。"""
    stack = [(path, False)]
    while stack:
        cur, processed = stack.pop()
        if processed:
            deleteFile(cur)
        else:
            stack.append((cur, True))
            for child in getChildrenPaths(cur):
                stack.append((child, False))

讲解策略

  1. 画一棵小树，明确"后序遍历"是关键 — 先删子节点再删父节点，否则父删了子还在就泄漏。
  2. 递归 vs 迭代 trade-off（这题被反复追问）： - 递归：代码短，但深目录会爆栈。 - 迭代用 stack：稳定但代码长，要标记"是否已处理子节点"。
  3. API 失败处理：deleteFile 失败时是 retry、跳过、还是抛？要主动问。
  4. 并发：如果有其他进程在加文件，最终一致性怎么保证？讨论"加锁" / "delete 期间禁止 write" / "标记 tombstone 后异步 GC"。
  5. 复杂度：O(N) 总节点数，每个节点一次 API call。

常见坑

  • 先删父再删子：导致逻辑错误甚至无限循环。
  • 不考虑 getChildrenPaths 失败：网络抖动很常见，要 retry + idempotent。
  • 递归在 200+ 深目录上 RecursionError。
  • 没问"是否删根"：API 语义可能是"删 path 自身"或"清空 path 下"，影响代码。
