Q05. Text Editor / Book Keeping System（LC 588/1166 思想）

来源：L4 SWE 2026 R1 · LC: LC 588 / 2296 🟡 变体 · 难度: Med-Hard

OOP Design Two Stacks Trade-off 🟡 变体 Med-Hard

题目概览

来源：Google L4 SWE Interview Experience 2026 R1 原描述：类似 leetcode 上的 text editor / book keeping system 类题，重点考 OOP design、数据结构 trade-off、内存考量、time/space complexity。

可能题型与思路

这类题目变体很多，最常见三种：

(a) Simple Text Editor (LC 2296) — 实现 addText / deleteText / cursorLeft / cursorRight。 (b) Design In-Memory File System (LC 588) — ls / mkdir / addContentToFile / readContentFromFile。 (c) Custom Book Keeping — 借阅/归还/查询，多种查询模式。

通用澄清问题

  • EN: What operations must be supported? / Expected QPS / scale? / Are there ordering / sorting requirements? / Multi-user (concurrency)? / Persistence required? / Optimize for read or write?
  • 中：要支持哪些操作？数据规模？是否要排序/有序遍历？是否多用户并发？要持久化吗？读多还是写多？操作的时间复杂度有没有要求？

🧠 整体思路

这是 OOP 设计题，先 clarify 再写代码。题型常见三种（先问面试官）： 1. Text Editor (LC 2296)：addText / deleteText / cursorLeft / cursorRight。 2. In-Memory File System (LC 588)：ls / mkdir / addContentToFile。 3. Book Keeping (借阅系统)：借/还/查询。

最高频的 Text Editor 用 "两个栈" 模型：

left_stack:  [a b c d]    | cursor |    right_stack: [h g f e]
                                                       ↑ top
  • 光标左侧的字符存在 left 里（栈顶 = 光标前一个）。
  • 光标右侧的字符倒序存在 right 里（栈顶 = 光标后一个）。
  • 所有操作都是 push/pop，摊还 O(1)。

为什么不用数组：插入/删除中间字符是 O(n)，整个 IDE 卡得不行。

🪜 思路拆解

Step 1 · clarify 阶段（必须做） - 支持哪些 API？是否大小写敏感？多用户？要持久化吗？ - 期望复杂度？（"现实文本编辑器" 暗示 amortized O(1)） - 这一步面经里特别表扬：clarifications 是 Hire 的关键。

Step 2 · 设计内部状态 - left：栈，存光标左边的字符。left[-1] 是光标紧贴左侧的字符。 - right：栈，存光标右边的字符（倒序，栈顶 = 光标紧贴右侧）。 - 字符串就是 "".join(left) + "".join(reversed(right))。

Step 3 · addText(text) - 做什么：在光标处插入 text。 - 实现：把 text 的字符依次追加到 left（push）。 - 复杂度：O(len(text))。

Step 4 · deleteText(k) - 做什么：删光标左边 k 个字符。 - 实现：从 left 末尾 pop k 个（不够就全 pop）。 - 返回值：实际删除的数量。

Step 5 · cursorLeft(k) - 做什么：光标向左移 k 步。 - 实现：从 left pop 一个 char → push 到 right。重复 k 次或到边界。 - 返回值：题目要求返回光标左侧最后 10 个字符（"".join(left[-10:])）。

Step 6 · cursorRight(k) - 做什么：光标向右移 k 步。 - 实现：从 right pop → push 到 left。 - 返回值：同 cursorLeft。

Step 7 · 持续 dry run - 每写完一个 API，举一个具体例子走一遍（比如 addText("hello") → cursorLeft(3) → deleteText(1) → cursorRight(5)）。 - Google 面经特别强调 "continuous dry runs"。

⌨️ 逐步实现

Step 2 · 初始化

class TextEditor:
    def __init__(self):
        self.left = []     # 光标左侧字符 (栈顶 = 最靠近光标)
        self.right = []    # 光标右侧字符, 倒序存放

Step 3 · addText

    def addText(self, text: str) -> None:
        self.left.extend(text)

Step 4 · deleteText

    def deleteText(self, k: int) -> int:
        k = min(k, len(self.left))
        del self.left[-k:] if k else None
        return k

Step 5 · cursorLeft

    def cursorLeft(self, k: int) -> str:
        while k and self.left:
            self.right.append(self.left.pop())   # 左 -> 右
            k -= 1
        return "".join(self.left[-10:])

Step 6 · cursorRight

    def cursorRight(self, k: int) -> str:
        while k and self.right:
            self.left.append(self.right.pop())   # 右 -> 左
            k -= 1
        return "".join(self.left[-10:])

Python 完整实现：Text Editor (LC 2296 风格)

class TextEditor:
    """O(1) amortized for all ops using two stacks split at cursor."""

    def __init__(self):
        self.left = []   # characters before cursor
        self.right = []  # characters after cursor (reversed for stack semantics)

    def addText(self, text: str) -> None:
        self.left.extend(text)

    def deleteText(self, k: int) -> int:
        k = min(k, len(self.left))
        del self.left[-k:]
        return k

    def cursorLeft(self, k: int) -> str:
        while k and self.left:
            self.right.append(self.left.pop())
            k -= 1
        return "".join(self.left[-10:])

    def cursorRight(self, k: int) -> str:
        while k and self.right:
            self.left.append(self.right.pop())
            k -= 1
        return "".join(self.left[-10:])

讲解策略

  1. 先 clarify 操作集：千万别上手就写，让面试官说清楚 API。
  2. 画 ADT 草图：把每个 op 的输入/输出列出来，让面试官点头确认。
  3. 数据结构候选：array (慢)、linked list (随机访问慢)、gap buffer / two stacks (经典文本编辑器实现) — 主动讲 trade-off。
  4. 持续 dry run：每个 API 写完先用例子走一遍。
  5. OOP 清洁：内部状态封装好，方法名一目了然，不要把 "private" 逻辑暴露出去。

常见坑

  • 直接写一个数据结构而不讨论替代：Google 评分明确包含 trade-off discussion，没讨论就丢分。
  • 没有 dry run：题目里就提到 "continuous dry runs" 是给 Hire 的关键。
  • 过早优化：先写干净的 O(N) 版本，让面试官提"能不能更快"再优化。
  • 忘记 edge case：空字符串、cursor 已经在边界、k 比剩余长度大。
